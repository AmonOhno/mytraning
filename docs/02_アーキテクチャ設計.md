# MyTraining アーキテクチャ設計書

## 1. 全体構成

```
┌─────────────────────────────────────────────┐
│                  共通コードベース (TypeScript)         │
│  React + Vite                                │
│  ├── UI コンポーネント (src/components)         │
│  ├── ドメインロジック (src/lib)                  │
│  └── ストレージ層 (localStorage)                │
└──────────────┬──────────────┬───────────────┘
               │              │
        Web ブラウザ      Capacitor (ios/)
        (そのまま配信)     → Xcode → App Store
               │              │
               └──────┬───────┘
                      ▼
              MyOS 連携 (JSON / Webhook)
```

## 2. 技術選定と理由

| 領域 | 採用技術 | 理由 |
|---|---|---|
| 言語 | TypeScript | 要件指定。型安全にドメインモデルを共有 |
| UI | React 19 + Vite | 高速な開発体験。Capacitor と相性が良い |
| iOS | Capacitor | Web 資産をそのまま App Store 配信形式にラップ。無料 OSS |
| 保存 | localStorage | サーバ不要=コスト無料。日次記録程度のデータ量なら十分 |
| ホスティング(Web) | GitHub Pages / Cloudflare Pages 等 | 静的サイトのため無料枠で運用可能 |

### 代替案と不採用理由
- **React Native / Swift ネイティブ**: コードベースが分裂し、TypeScript Web 版との二重管理になるため不採用。
- **Firebase / Supabase**: 無料枠はあるが外部依存・将来課金リスクがあるため初期は不採用。データモデルは将来移行可能な形で設計。

## 3. レイヤ構成

```
src/
├── types.ts          # ドメイン型定義(記録・セット・設定)
├── lib/
│   ├── storage.ts    # localStorage の読み書き(永続化層)
│   ├── stats.ts      # 週間サマリー等の集計ロジック
│   └── myos.ts       # MyOS 連携(エクスポート・Webhook)
├── components/       # 画面コンポーネント
│   ├── RecordForm.tsx    # 記録入力・編集
│   ├── RecordList.tsx    # 記録一覧
│   ├── Summary.tsx       # 週間サマリー
│   └── Settings.tsx      # MyOS 連携設定・エクスポート
└── App.tsx           # タブナビゲーションとルーティング
```

依存方向: `components → lib → types`。UI からストレージへ直接アクセスしない。

## 4. データフロー

1. ユーザーが `RecordForm` で入力 → バリデーション → `storage.saveRecord()`
2. `storage` は localStorage キー `mytraining.records` に JSON 配列で保存
3. 一覧・サマリーは保存済み配列から導出(単方向データフロー)
4. MyOS 連携時は `myos.ts` が連携フォーマットへ変換して出力

## 5. iOS ビルドパイプライン

```
npm run build          # Web 資産を dist/ に生成
npx cap sync ios       # dist/ を iOS プロジェクトへコピー
npx cap open ios       # Xcode で開き、署名してアーカイブ → App Store Connect
```

詳細は「05_iOS_AppStore配信手順.md」を参照。
